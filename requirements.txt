## 1. Application vision

VAgeWell Care requires a unified internal healthcare deployment platform that carries a patient and their designated dependents through the continuous home-care delivery loop, built as a progression of modules inside a single platform interface. It begins the instant an account is **initialized** via verification, with each stage's outputs acting as the foundational data layer for downstream operations.

* **V1 — Single identity hub, cascading profiles.** All clinical, operational, and administrative workflows run inside one application framework. They share a primary account record linked to individual dependent sub-profiles, ensuring single-sign-on access for entire multi-generational households.
* **V2 — Active at intake.** Cold inquiries, promotional outbox logs, and lead capture pipelines are permanently out of scope. The platform's operational boundary begins precisely at the moment a registration token is successfully confirmed via SMS.
* **V3 — Foundation first.** The initial intake engine consolidates structural demographic markers, medical histories, and relational family parameters into a verified foundation file. Downstream features (scheduling grid, on-site diagnostics) look up this master file rather than re-interviewing the patient.
* **V4 — Care staff as the platform's eyes and legs.** Where physiological data must be gathered from physical spaces (vitals tracking, medication compliance tracking, chronic symptom audits), the application relies on secure staff data entries collected directly at the point of care during live home visits.

**Application-level out of scope:** Complex medical diagnoses, remote prescription fulfillment engines, automated external third-party pharmacy integrations, corporate vendor onboarding logs.

## 2. Module 1 — Care Intake & Verification Foundation

**Purpose.** Aggregate initial patient information, establish family group hierarchies under shared credentials, map dynamic multi-day services to transactional ledgers, clear financial processing states, and provide a secure, human-verified gateway for downstream care distribution.

### R1 — Inputs the module must ingest

| # | Input | Notes |
| --- | --- | --- |
| R1.1 | Cryptographic OTP token | Client-side entries matching the 6-digit verification code generated by the connected SMS gateway provider |
| R1.2 | Master account metadata | Structural demographic metrics: Full Name, Primary Contact Phone, Age, and Gender parameters |
| R1.3 | Dependent relationship metrics | Secondary profile files including Name, Age, specific Relationship tags, and individual contact numbers |
| R1.4 | Operational scheduling values | The user's targeted care selections, requested calendar date, multi-day assignment length, and target time slot configurations |
| R1.5 | Symptom/Problem briefings | Textual self-reported logs mapping current physical struggles, chronic concerns, or localized pain markers |
| R1.6 | Payment validation screenshots | Image uploads (`payment.jpg`) capturing the completed transaction screen from the customer's personal UPI payment app |
| R1.7 | On-site clinical metrics | Real-time diagnostics (Blood Pressure, Glucose levels, SpO2 tracks) entered manually by clinical staff during home check-ins |

### R2 — Outputs the module must produce (the foundation record)

| # | Output | Notes |
| --- | --- | --- |
| R2.1 | Authenticated JWT session | Valid runtime tokens linking the current interface context cleanly to the primary relational data tables |
| R2.2 | Profile relationship matrix | Structural linkage mapping the main client row down to secondary rows inside the `family_members` table |
| R2.3 | Evaluated billing matrix | Dynamic fee evaluation logic parsing active care tracks against standard daily multipliers (e.g., Physiotherapy at ₹1500/day) |
| R2.4 | Aggregate financial calculation | A real-time calculated balance string: $\text{Days} \times \text{Price Per Day} = \text{Total Amount}$ |
| R2.5 | Transactional status flags | Enforced operational identifiers restricted to four states: `pending`, `pending_verification`, `paid`, or `pay_at_visit` |
| R2.6 | Asynchronous operations digest | Structured email payload assembled automatically via standard endpoints to alert operational management |
| R2.7 | Protected clinical ledger | Isolated profile matrices tracking blood pressure histories, metabolic fluctuations, and underlying systemic diagnoses |

### R3 — Behaviors the module must have

* **R3.1 Role validation over UI navigation.** Bypassing interfaces via hidden client layouts yields zero structural permissions. The system must natively check the authenticated profile's `role` column against the server database to confirm `admin` or `leaf_node` privileges before exposing workspace tools.
* **R3.2 Relational proxy management.** The account manager serves as the shared transaction proxy. The platform must allow the registration of dependents without forcing profile logouts or requiring secondary account creation, while still maintaining unique sub-profile contact records.
* **R3.3 Manual financial quarantine.** Selecting an online payment track locks down the booking lifecycle to `pending_verification`. The status must remain frozen until an administrative profile visually validates the screenshot proof and overrides the flag to `paid`.
* **R3.4 Proactive triage distribution.** The exact instant a new schedule request or payment receipt is committed, the interface must fire a structured notification package tracking clinical metrics directly to the target mailbox (`maheshwari21102003@gmail.com`).
* **R3.5 Data validation gating.** Downstream workflows (such as scheduling field assignments or chronic symptom trending analytics) must remain locked behind a completeness gate until the base profile entry is verified.

### R4 — Constraints and rules

* **R4.1 Architectural dependency.** Built natively for relational storage patterns (Supabase PostgreSQL). Custom attributes must extend the structural integrity of the baseline `profiles` and `family_members` tables rather than living in disparate client arrays.
* **R4.2 Asymmetric field permissions.** Standard user profiles retain full permissions to modify baseline biographical values (Name, Age). However, vital registers (Blood Pressure, Sugar Level) are read-only to the patient and mutable only by profiles cleared through `public.is_staff()` policies.
* **R4.3 Row-level isolation mandate.** Patient accounts are structurally restricted to selecting and mutating rows matching their active `auth.uid()`. Internal staff overrides bypass these restrictions via server-validated security policies.

## 5. Gap Resolutions — planning clarification (Jul 20, 2026)

*Answers to the 7 open gaps in `requirements-detail-prompt.md` (IDs A–H), captured from the Jul 20, 2026 clarification. Each item is marked **RESOLVED** or **OPEN**. Open items must be answered before their build step — they are NOT to be guessed. Resolutions here supersede the referenced R-items where they refine them.*

### 5A — Resolved

* **GR-1 — Unified OTP authentication (A1, A2, A3, C1).** All users — patients, coordinators, staff, and admins — authenticate through one identical gateway: phone number + **6-digit Twilio SMS OTP** (Supabase Auth, `auth.uid()`). There is no username/password path and no separate admin login screen. Every login is a fresh OTP; no password is ever set on the account. → refines R1.1, R2.1, R3.1; confirms Discovery Q3 (Twilio selected over MSG91).
* **GR-2 — 72-hour session lifetime (A4).** A verified session stays valid for **72 hours** before re-authentication is required. → refines R2.1. Note: this overrides the platform default of an 8-hour session; the session/token expiry for this project is 72 hours.
* **GR-3 — Billing math is flat multiplication (B3).** Total = **Days Requested × Price Per Day**, with **no** add-ons, discounts, taxes/GST, or minimum-booking surcharges. → confirms R2.3, R2.4.
* **GR-4 — Unrestricted scheduling (F1, F2).** Patients pick any date and any **15-minute** time slot purely by need. The system only **records** the requested date / duration / slot — there is **no** calendar blocking, nurse-availability check, or double-booking prevention in Module 1. → refines R1.4.
* **GR-5 — Dual-channel payment gate (G3).** Two channels: **Direct Payment → `pay_at_visit`** (settled in person; no proof upload) and **Online UPI → `pending_verification`** (patient uploads a UPI payment screenshot). Online bookings stay frozen at `pending_verification` until an admin visually validates the screenshot and manually overrides the flag to `paid`. → confirms R2.5, R3.3.
* **GR-6 — Admin-only clinical ledger (E2 partial, H1 partial).** The clinical ledger — **Blood Pressure, Blood Group, Sugar Level, and Medical Conditions** — is filled and modified **only** by an administrator/staff, and is read-only to the patient. → refines R2.7, R4.2 (adds Blood Group and Medical Conditions to the previously named Blood Pressure / Sugar Level).
* **GR-7 — Booking-finalization notification (D3).** A structured email notification fires automatically to `maheshwari21102003@gmail.com` **the instant a booking is finalized**. → confirms R3.4.

### 5B — Still open (answer before the noted step; do not guess — Rule 4)

* **GO-1 — Full service catalog & pricing (B1, B2, B4).** Only "Physiotherapy — ₹1500/day" is known. Need every bookable service, its per-day price (flat or variable), and whether one booking may include multiple services or is one-service-per-booking.
* **GO-2 — Notification channel & exact payload (D1, D2).** Delivery channel not named — plain email (SMTP/SendGrid) vs the internal Zoho Cliq relay. The exact field list of the "structured payload" was asserted but never enumerated. Inferred candidate set for confirmation: patient name, service(s), date(s), days, price/day, total amount, payment status, symptom brief.
* **GO-3 — Role model (E1, E3).** The document mixes `admin` / `leaf_node` (R3.1) with Patient/Coordinator, Admin, and Staff (Discovery Q2). Need the exact set of `role` column values and whether Admin, Staff, and `leaf_node` are distinct roles or aliases.
* **GO-4 — OTP operational rules (C2, C3).** OTP expiry window, resend/retry limits, and dev/test behavior (real SMS send vs a fixed test code) are unspecified.
* **GO-5 — Payment screenshot storage & rejection (G1, G2, G3).** Storage target (Supabase Storage vs GCS), max file size, allowed MIME types, and the admin-rejection transition (does a rejected screenshot go back to `pending`, stay `pending_verification`, or a new state?) are unspecified.
* **GO-6 — Multi-day shape (F3).** Whether "multi-day assignment length" is a continuous run of consecutive days or allows non-consecutive dates is unspecified.
* **GO-7 — Dependents & data governance (H2, H4).** Max dependents per account, any required dependent fields beyond Name/Age/Relationship/Contact, and any medical-data retention/consent rules to enforce now are unspecified. (H3 — city/region restriction — is deferred to a future module; no restriction applies in Module 1.)

### 5C — Conflict flagged (Rule 5 — No Silent Substitution)

* **CF-1 — Auth model diverges from the platform's shared auth system.** The platform default is username/password auth against the shared `employees` / `apps` tables (bcrypt, HS256 JWT). VAgeWell instead requires **Supabase Auth phone + OTP with `auth.uid()` + row-level security** for its own patient/staff users (R3.1, R4.3, Discovery Q4). These are two different auth systems. This requires an explicit decision before Step 1 — it must not be silently reconciled either way.
